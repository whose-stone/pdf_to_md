# pdf_to_md

A command-line tool that converts PDFs into readable Markdown. Text layout is reconstructed with heading/list heuristics, and embedded images are replaced with LLM-generated descriptions via any OpenAI-compatible chat completions endpoint (cloud or self-hosted).

## Features

- Page-by-page extraction with `<!-- Page N -->` separators.
- Heading and list detection based on font-size heuristics.
- Image regions replaced with `![alt](description)` lines generated by a vision-capable LLM.
- Optional LLM post-processing pass to clean up layout artifacts.
- Fully offline mode via `--no-llm`.
- Works with any OpenAI-compatible API (OpenAI, Azure OpenAI, vLLM, llama.cpp, LM Studio, Ollama's OpenAI shim, etc.).

## Requirements

- Python 3.11+ (uses stdlib `tomllib`; falls back to `tomli` on older versions).
- Dependencies: `PyMuPDF`, `requests`, `python-dotenv` (see `requirements.txt`).

## Install

```bash
# Windows (PowerShell)
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

# macOS / Linux
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Configuration

Configuration is split between a TOML config file (non-secret settings) and environment variables (the API key).

### Interactive setup

The `runme/` directory contains scripts that prompt for every setting and write both `config.toml` and `.env` for you:

```bash
# Windows (PowerShell)
.\runme\setup.ps1

# macOS / Linux
./runme/setup.sh
```

See `runme/README.md` for details. The sections below document the files these scripts produce.

### `config.toml`

Resolution order:
1. `--config <path>` flag.
2. `$XDG_CONFIG_HOME/pdf_to_md/config.toml` (Linux/macOS) or `~/.config/pdf_to_md/config.toml`.
3. Windows: set `XDG_CONFIG_HOME` or pass `--config`. If no config file is found, the tool falls back to environment variables (`PDF_TO_MD_MODEL`, `PDF_TO_MD_BASE_URL`, `LLM_API_KEY`).

Example (`examples/config.toml`):

```toml
[llm]
base_url = "https://api.openai.com/v1"
model = "gpt-4.1-mini"
api_key_env = "LLM_API_KEY"   # name of the env var that holds the key
temperature = 0.0
max_tokens = 800
enabled = true
describe_images = true
postprocess_layout = true
```

`api_key_env` is the **name of the environment variable** to read the key from — not the key itself. Keep secrets out of the TOML file.

### `.env` file

On startup the tool calls `dotenv.load_dotenv()`, so any variables defined in a `.env` file in the current working directory are loaded into the process environment automatically. This is the recommended way to provide the API key during development.

```bash
cp .env.example .env
# then edit .env
```

`.env.example`:

```
LLM_API_KEY=your-api-key-here
```

The variable name must match `api_key_env` in `config.toml` (default `LLM_API_KEY`). `.env` is gitignored. In production, prefer exporting the variable through your process manager or secret store rather than shipping a `.env` file.

For local inference endpoints (`localhost` / `127.0.0.1`) the API key check is skipped.

## Usage

Single file:

```bash
python pdf_to_md.py path/to/input.pdf
```

Batch mode — drop PDFs into `input/` and run with no arguments; converted markdown is written to `output/` using the same basename:

```bash
python pdf_to_md.py
```

Flags:

| Flag | Description |
| --- | --- |
| `-o, --output PATH` | Output Markdown path (single-file mode). Defaults to the input path with `.md` extension. |
| `--input-dir PATH` | Batch-mode input directory. Defaults to `./input`. |
| `--output-dir PATH` | Batch-mode output directory. Defaults to `./output`. |
| `--config PATH` | Path to a TOML config file. |
| `--no-llm` | Skip all LLM calls (image description and layout post-processing). |

Exit codes: `0` success, `1` on missing input or conversion failure.

## Building a standalone binary

```bash
pip install pyinstaller

# CLI binary
pyinstaller --clean --noconfirm pdf_to_md.spec

# Windows GUI binary (single-file, no console window)
pyinstaller --clean --noconfirm pdf_to_md_gui.spec
```

Both specs emit a single-file binary into `dist/`. On Windows this produces `dist/pdf_to_md.exe` and `dist/pdf_to_md_gui.exe`.

## GUI

`pdf_to_md_gui.py` (and the packaged `pdf_to_md_gui.exe`) provides a Tkinter interface for users who don't want to work from the shell:

- Browse for an **input folder** (directory of PDFs) and an **output folder** (where `.md` files will be written).
- Set the **API Base URL**, **API Key**, **Model**, **Temperature**, and **Max tokens**.
- Toggle **Enable LLM / Describe images / Postprocess layout**.
- Click **Save profile** to persist settings so they're restored next launch.
- Click **Convert** to batch-convert every PDF in the input folder; progress is streamed to the log panel.

### Profile storage

Settings are written to a local JSON file:

- **Windows:** `%APPDATA%\pdf_to_md\profile.json`
- **macOS / Linux:** `~/.config/pdf_to_md/profile.json`

The file contains the API key in plain text — treat it the same way you would a `.env` file. It lives entirely on the user's machine and is never bundled into the EXE.
